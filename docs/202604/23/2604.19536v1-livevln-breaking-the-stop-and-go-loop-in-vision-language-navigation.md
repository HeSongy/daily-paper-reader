## 摘要
近年来的导航系统在基准测试中表现出色，但在实际部署中仍常出现明显的“停-走”现象。造成这一瓶颈的原因在于其感知—推理—执行循环依然是阻塞式的：每次新的观测后，控制器必须等待感知、传输与推理完成才能继续运动。因此，仅降低动作生成成本并不能消除多余的等待。为解决这一问题，我们提出了 LiveVLN，这是一种无需训练的框架，通过为预训练的 VLM 导航器增加多步动作延续，实现更连续的具身导航。LiveVLN 不再在每次完整的感知与推理循环中暂停，而是将执行过程与新观测的处理部分重叠，使得在当前可执行动作序列耗尽之前，即可交付更新的未来动作。此设计在运动过程中持续保持动作可用性，减少空闲等待，使在线执行更加流畅。该框架在运行时工作，可与兼容的预训练 VLM 导航器集成。在 R2R 和 RxR 数据集上，LiveVLN 保持基准性能的同时，显著降低等待时间并提高动作可用性。在真实环境部署中，平均单回合等待时间减少最高达 77.7%，在 StreamVLN 上总回合时间缩短 12.6%，在 NaVIDA 上缩短 19.6%，从而实现更一致的执行效果。代码可在 https://github.com/NIneeeeeem/LiveVLN 获取。

## Abstract
Recent navigation systems achieve strong benchmark results, yet real-world deployment often remains visibly stop-and-go. This bottleneck arises because the sense-inference-execution loop is still blocking: after each new observation, the controller must wait for sensing, transmission, and inference before motion can continue. Reducing action-generation cost alone therefore does not remove redundant waiting. To address this issue, we present LiveVLN, a training-free framework for more continuous embodied navigation by augmenting pretrained VLM navigators with multi-step action continuation. Instead of pausing for each full sense-and-inference round, LiveVLN overlaps execution with the processing of newly arrived observations, allowing refreshed future actions to be handed off before the current executable prefix is exhausted. This design keeps actions continuously available during motion, reducing idle waiting and enabling smoother online execution. The framework operates at runtime and can be integrated with compatible pretrained VLM navigators. Across R2R and RxR, LiveVLN preserves benchmark performance while reducing waiting time and improving action availability. In real-world deployments, it cuts average episode waiting time by up to $77.7\%$ and shortens wall-clock episode time by $12.6\%$ on StreamVLN and $19.6\%$ on NaVIDA, yielding more coherent execution during deployment. Code is available at https://github.com/NIneeeeeem/LiveVLN.

## 一、研究问题与动机

### 1. 问题背景
- **视觉语言导航（Vision-Language Navigation, VLN）** 旨在让机器人基于自然语言指令与实时视觉输入自主行动，如“走到窗户旁的椅子旁”。
- 许多现代方法（如 **VLN-BERT、DUET、NaVIDA、StreamVLN** 等）在离线基准（如 R2R、RxR）上表现优异，但在**真实部署**中常出现“停-走”（stop-and-go）现象。
- 导致该问题的根源不是模型性能，而是**运行时结构性阻塞**：
  - 普通流程：感知（sense）→ 推理（inference）→ 执行（execution）是**串行的**。
  - 每次生成动作后需等待下一轮感知与推理完成，机器人因此反复停顿。

### 2. 研究动机
- 仅优化推理速度无法根除停顿，因为延迟来自系统的接口结构。
- 实际部署中，机器人约 **30% 以上时间处于等待状态**（例如 NaVIDA 平均等待 10 秒 / 回合）。
- 因此，作者希望设计一种**训练无关（training-free）**的运行时框架，实现：
  - 动作生成与感知推理**并行化**；
  - 在动作执行时就预先刷新下一步计划；
  - 保持导航性能同时减少等待延迟。

## 二、方法论：LiveVLN 框架

### 1. 核心思想
- **LiveVLN** 是一种在运行时附加于已有 VLM 导航器的训练自由框架。
- 它**打破 sense–inference–execution 的阻塞结构**，通过**双线程异步机制**实现执行与推理的重叠：
  - 线程 A：持续执行当前动作序列；
  - 线程 B：在后台接收新观测并生成更新的动作延续。

### 2. 关键机制
#### （1）短时行动状态（short-horizon action state）
系统维护三部分：
- **已执行动作 (executed actions)**：已完成部分。
- **守卫缓冲 (guard buffer)**：当前正在执行的短行动序列，可保证连续性。
- **可修订尾部 (revisable tail)**：未公开的未来动作，可在下一次推理中被替换。

#### （2）守卫交接（Guarded Handoff）
- 当后台推理完成时，如果在当前 guard buffer 尚未执行完之前：
  - 当前缓冲区转入已执行部分；
  - 刷新前缀成为新的 guard buffer；
  - 尾部继续保留为修订区。
- 确保控制器在刷新过程中**始终有动作可执行**。

#### （3）实时自适应（Real-Time Adaptation）
- 实时测量上一轮推理延迟（ℓₜ,SI）并平滑估计未来延迟：
  - 更新规则：  
    \[
    \tilde{\ell}_{t+1,SI} = (1-\alpha)\tilde{\ell}_{t,SI} + \alpha \ell_{t,SI}
    \]
  - 设置安全裕度 δ，计算下一轮守卫时长 ψₜ₊₁ = 𝑙̃ₜ₊₁,SI + δ。
- 动态选择最短的动作前缀长度 k* 使得执行时间 ≥ ψₜ₊₁。
- 若推理未能在守卫执行完前完成，则可执行“备选动作”或执行停止，确保安全性。

#### （4）算法流程概述
1. 持续执行当前守卫缓冲；
2. 后台刷新新动作延续；
3. 估计下一轮推理延迟；
4. 动态调整下一守卫长度；
5. 若延迟满足条件则完成交接，否则执行补救动作。
